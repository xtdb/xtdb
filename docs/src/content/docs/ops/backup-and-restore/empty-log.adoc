---
title: Scenario - Empty Log & Intact Storage
---

If the log is unexpectedly empty — for example, due to topic deletion or reinitialization — but previously indexed storage still exists, XTDB will refuse to start. 
This is a safety mechanism designed to prevent state divergence between the log and the object store.

You may encounter an error such as:

> "*Node failed to start due to an invalid transaction log state (the log is empty) that does not correspond with the latest indexed transaction (epoch=0 and offset=289).*"

== What Is Preserved

If the storage backend remains intact, then all previously compacted and indexed data is still available.

* Historical data in all compacted Arrow files (Level-0 and higher)
* The full queryable state of the database, up to the last successfully indexed offset

== What Is Lost

Any recently submitted transactions that were not indexed prior to the loss of the transaction log:

* These are unrecoverable unless they were backed up separately or stored externally.

== Recovery Strategy: Using Epochs

XTDB supports a mechanism called `epoch` to intentionally reset log offset tracking. This allows you to reinitialize the log while retaining and continuing from the existing storage snapshot.

When configured with a new epoch, the node treats the current storage state as canonical and begins writing to the empty log as if from scratch.

== Recovery Steps

1. **Determine the current epoch value**  
   See the contents of the error message for the current epoch. This is typically `0` for a new installation.

2. **Choose a new `epoch` value.**  
   Select an integer greater than the previously used epoch value for this log - generally, `epoch + 1`.

3. **Update the node configuration:**  
+
[source,yaml]
----
log: !<LogType>
  epoch: <new-epoch>
----

4. **Restart the node.**  
   XTDB will skip offset validation and begin writing to the empty topic under the new epoch.

5. **(Optional) Re-submit any lost transactions.**  
   If you are aware of missing transactions, reissue them manually or via your application’s recovery layer.

== Cluster-Wide Considerations

* All nodes in the cluster **must** use the same `epoch` value. Mismatches will cause startup failures.
* It is strongly recommended to apply the epoch change atomically:
1. Shut down all nodes
2. Update configuration
3. Restart nodes together

== Notes & Warnings

* Always take a backup of storage before making epoch changes.
* The new epoch marks a clean slate for the log but does **not** delete existing data in the object store.
* Ensure the log backend (e.g. Kafka) is writable and healthy before restarting.
* Lost log messages are non recoverable unless previously exported or replicated elsewhere.
