# 5054 – TxSink initial scan from L0 files

Related issue(s): GH-5054
Source of truth for task status: the external issue tracker.

---

## Task Summary

Add a config option to TxSink that enables an "initial scan" of L0 files when no message is found on the output topic. This allows TxSink to output all historical transactions on first startup, rather than only outputting transactions from the current indexer position forward.

This is part of a larger feature to allow TxSink to restart from where it left off. The next stage will handle catching up from an arbitrary offset (when the last output message is *before* the current indexer offset).

---

## Current Understanding

**Goal**: When `initialScan: true` and no message exists on the output topic, scan L0 files and output all historical transactions.

**L0 file structure (to be verified)**:

* L0 files contain raw events in a trie structure
* Leaves contain ≤1024 events, keyed by iid prefix
* Events within a batch are ordered by system time
* Block index is in system time order
* All tables that exist for a given block index have a data file written

**Resolved questions**:

1. **Is a file written for a table that has no events in a given time period?**
   - No - only tables with events in a block get L0 files written.

2. **Do all files for a block index share the same system time range?**
   - Yes - all events flushed in one `flush-block!` call share the same block index.

3. **How do we identify which events belong to the same transaction?**
   - Events with the same `_system_from` are from the same transaction. The tx-id comes from `xt.txs` table.

4. **How exactly is the trie structure organized?**
   - See `LiveTrieWriter.kt` for implementation details.

5. **One trie-key per table per block-idx?**
   - Yes, should be enforced. Add assertion in `build-l0-index`.

6. **Raw events or resolved?**
   - Raw events, translated to same output format as TxSink. Reuse utils/classes from existing code.

7. **Output grouping?**
   - Per transaction (grouped by `_system_from`), not per block.

8. **Output format construction?**
   - tx-key from `xt.txs`, system_time from any event, db-name from startup, block-idx known, table ops from events.

**Design constraints**:

* Code should be specific to this stage but make the next stage (starting from arbitrary offset) easy to implement
* Need to handle tables being created dynamically (not all tables exist from start of time)
* Should minimize memory usage by only opening L0 files for one block index at a time

## Important Learnings

### L0 File Structure
- L0 files are written per-table per-block (one L0 file per table per `flush-block!`)
- File naming: `l00-rc-b<block-idx>` (level 0, current recency, block index in lex hex)
- Data files are Arrow format, stored at `tables/<table>/data/<trie-key>.arrow`
- Meta files at `tables/<table>/meta/<trie-key>.arrow`
- Data file schema includes: `_iid`, `_system_from`, `_valid_from`, `_valid_to`, `op` (union of put/delete/erase)
- Events in L0 are NOT deduplicated for same system-time - the `BufferPoolSegment` resolves these when `level == 0`

### Block Index Ordering
- Block files stored at `blocks/b<idx>.binpb` (protobuf format)
- `BlockCatalog.allBlockFiles` returns all block files sorted by name (lex hex ensures correct ordering)
- Each block file contains: `blockIndex`, `latestCompletedTx`, `latestProcessedMsgId`, `tableNames`
- Block index increments on each `flush-block!`

### Trie Structure
- `MemoryHashTrie` uses branches (16-way by iid nibble) and leaves (≤1024 events)
- Leaves contain row indices into the data relation
- Events within a leaf are sorted by iid first, then by row index (newer events first for same iid)
- L0 files have `level == 0` and need same-system-time resolution when reading

### TxSink Current Behavior
- Reads `last-tx-key` from output log on startup via `readLastMessage`
- On each commit, compares `tx-key` against `last-tx-key` - only outputs if newer
- Output format: `{:transaction {:id tx-key}, :system-time, :source {:db, :block-idx}, :tables [...]}`
- Each table entry: `{:db, :schema, :table, :ops [{:op, :iid, :valid-from, :valid-to, :doc}]}`

### Key Insight for Implementation
- To read all events from L0 files in system-time order across tables:
  1. Get all L0 trie keys from trie catalog (filter by level 0)
  2. Sort by block-idx to process in order
  3. For each block-idx, load all table L0 files for that block
  4. Events with same `_system_from` belong to same transaction
  5. Use `xt.txs` table entries to get transaction metadata

### Catch-up Loop Design
The initial scan must use a **catch-up loop**:
```
loop:
  current_block_idx = last block we've processed (or -1)
  refresh trie catalog / block catalog from object store
  available_blocks = get L0 block indexes from trie catalog
  blocks_to_process = available_blocks > current_block_idx

  if blocks_to_process is empty:
    switch to live-tail mode (existing behavior)
    break
  else:
    for each block in blocks_to_process:
      read L0 files for that block
      output transactions
    goto loop  # re-check for new blocks
```

### Constraints
- **L0 files are never deleted** - safe to rely on them existing on disk
- **L0 files CAN be marked as garbage** - must read from `:live` AND `:garbage` states
- **Block files CAN be GC'd** - must NOT rely on block files for enumeration
- Instead, enumerate block indexes from L0 trie keys in the trie catalog

### Enumerating L0 Block Indexes

See "Design Options" section below for three approaches being evaluated.

Key helper to get L0 tries (both live AND garbage) for a table:
```clojure
(defn l0-tries-for-table [trie-cat table]
  (let [{:keys [live garbage]} (-> (trie-cat/trie-state trie-cat table)
                                    (get-in [:tries [0 nil []]]))]
    (concat live garbage)))
```

To refresh and get latest state from object store:
```clojure
(.refresh trie-cat)
```

### Reusable L0 Reading Interface (Option A: Index + Reader)

Two separate, composable pieces:

**1. Index builder** - returns sorted map of `block-idx -> [{:table :trie-key}]`:

```clojure
(defn l0-tries-for-table [trie-cat table]
  (let [{:keys [live garbage]} (-> (trie-cat/trie-state trie-cat table)
                                    (get-in [:tries [0 nil []]]))]
    (concat live garbage)))

(defn build-l0-index [trie-cat]
  ;; Returns sorted map: {0 [{:table t1 :trie-key k1} {:table t2 :trie-key k2}], ...}
  ;; Asserts one trie-key per table per block-idx
  (->> (.getTables trie-cat)
       (mapcat (fn [table]
                 (->> (l0-tries-for-table trie-cat table)
                      (map (fn [{:keys [block-idx trie-key]}]
                             [block-idx {:table table :trie-key trie-key}])))))
       (group-by first)
       (map (fn [[block-idx pairs]]
              (let [entries (mapv second pairs)]
                ;; Assert one trie-key per table per block
                (assert (= (count entries) (count (distinct (map :table entries))))
                        (str "Multiple trie-keys for same table in block " block-idx))
                [block-idx entries])))
       (into (sorted-map))))
```

**2. Block reader** - reads events for a specific block:

```clojure
(defn read-l0-block [buffer-pool block-idx table-entries]
  ;; Returns seq of {:table TableRef, :events [...raw arrow rows...]}
  ;; Reuses existing utils/classes where possible
  )
```

**Usage in TxSink**:

```clojure
(doseq [[block-idx table-entries] (subseq (build-l0-index trie-cat) > last-block)]
  (let [block-events (read-l0-block buffer-pool block-idx table-entries)
        txs (group-events-by-tx block-events)]
    (doseq [tx txs]
      (output-tx! output-log tx))))
```

This design allows reuse for future tools (e.g., L0 transformation/copy tool) that need to iterate L0 files in order.

Usage in catch-up loop (using `subseq` for efficient filtering):

```clojure
(loop [last-processed-block -1]
  (.refresh trie-cat)
  (let [block->tables (build-l0-index trie-cat)
        blocks-to-process (subseq block->tables > last-processed-block)]
    (if (empty? blocks-to-process)
      :switch-to-live-mode
      (do
        (doseq [[block-idx tables] blocks-to-process]
          (process-block block-idx tables))
        (recur (first (last blocks-to-process)))))))
```

Test sketch:

```clojure
(t/deftest test-l0-index-building
  (with-open [node (start-node)]
    (jdbc/execute! node ["INSERT INTO foo RECORDS {_id: 1}"])
    (tu/flush-block! node)
    (jdbc/execute! node ["INSERT INTO bar RECORDS {_id: 1}"])
    (tu/flush-block! node)

    (let [block->tables (build-l0-index trie-cat)]
      ;; Sorted map, iteration is in block order
      (t/is (= [0 1] (keys block->tables)))
      ;; Block 0 has xt.txs + foo
      (t/is (= 2 (count (get block->tables 0))))
      ;; Block 1 has xt.txs + foo + bar
      (t/is (= 3 (count (get block->tables 1)))))))
```

### Pause/Resume for Testing: Inject process-block function

Pass `process-block-fn` as parameter, tests can wrap the default to add pause behavior:

```clojure
;; Production code takes process-block as parameter (default provided)
(defn default-process-block [block-idx tables output-log]
  ;; ... read L0 files, output transactions ...
  )

(defn catch-up-loop [trie-cat output-log {:keys [process-block-fn] :as opts}]
  (let [process-block (or process-block-fn default-process-block)]
    (loop [last-processed-block -1]
      (.refresh trie-cat)
      (let [block->tables (build-l0-index trie-cat)
            blocks-to-process (subseq block->tables > last-processed-block)]
        (if (empty? blocks-to-process)
          :switch-to-live-mode
          (do
            (doseq [[block-idx tables] blocks-to-process]
              (process-block block-idx tables output-log))
            (recur (first (last blocks-to-process)))))))))
```

Test sketch - wrap the processor to add pause behavior:

```clojure
(t/deftest test-catch-up-pause-resume
  (let [pause-after-block (promise)
        resume (promise)
        catch-up-done (promise)
        wrapped-processor (fn [block-idx tables output-log]
                            (default-process-block block-idx tables output-log)
                            (when (= block-idx 0)
                              (deliver pause-after-block true)
                              @resume))]
    (with-open [node (start-node)]
      (jdbc/execute! node ["INSERT INTO foo RECORDS {_id: 1}"])
      (tu/flush-block! node)

      (future
        (catch-up-loop trie-cat output-log {:process-block-fn wrapped-processor})
        (deliver catch-up-done true))

      (deref pause-after-block 5000 :timeout)

      (jdbc/execute! node ["INSERT INTO bar RECORDS {_id: 1}"])
      (tu/flush-block! node)

      (deliver resume true)

      (deref catch-up-done 5000 :timeout)
      (t/is (= 2 (count-output-messages output-log))))))
```
